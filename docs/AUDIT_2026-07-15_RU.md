# Аудит продукта и технический план

Дата: 15 июля 2026 года.

## Итоговая оценка

Проект уже решает основную задачу и заметно сильнее обычного «бота-формы»:
Telegram используется как быстрый мобильный интерфейс, а Google Sheets остаётся
понятным финансовым архивом. После выполненных исправлений общая готовность к
личному использованию — **8,3/10**. До уверенного unattended-production остаются
интеграционные тесты Telegram FSM, полная проверка batch-запросов на копии книги
и структурированная наблюдаемость.

| Область | Оценка | Вывод |
| --- | ---: | --- |
| Пользовательский сценарий | 8,5/10 | Есть назад, отмена, восстановление черновика и onboarding первого счёта. |
| Telegram UX и тексты | 8,3/10 | Единая панель и явное состояние превышенного бюджета; сложные списки пока без пагинации. |
| Финансовая модель | 8,5/10 | Разделены цена покупки и фактическое списание, сохранены исторические курсы. |
| Google Sheets | 8,2/10 | Append журнала идемпотентен и покрыт контрактными тестами сетевой неопределённости. |
| Надёжность | 8,5/10 | Есть атомарный undo, retry `Money`, operation ID, locks и безопасная компенсация replace. |
| Безопасность | 8/10 | После аудита доступ закрыт по умолчанию, state-файл создаётся с `0600`. |
| Тестирование | 7/10 | 56 тестов, включая gateway и recovery; пока нет полноценного Telegram FSM harness. |
| Эксплуатация | 7,5/10 | Добавлены `/status`, healthcheck, backup/recovery и debounce; метрик пока нет. |

## Что сделано во время аудита

1. Публичный доступ больше не включается случайно. Нужен allowlist или явный
   `ALLOW_PUBLIC_ACCESS=true`.
2. Ошибочные Telegram user ID больше не игнорируются молча.
3. `state.json` и временный файл создаются с правами `0600`.
4. Одна неудачная запись на диск больше не ломает навсегда очередь сохранения
   состояния.
5. Права существующих локальных `.env`, `credentials.json` и `data/state.json`
   ужесточены с `0644` до `0600` без чтения их содержимого.
6. Исправлены расхождения `.env.example`: удалена неиспользуемая переменная и
   выровнен default OpenAI model.
7. Добавлены тесты политики доступа, файловых прав и восстановления очереди.
8. Append расходов, пополнений и переводов стал идемпотентным по стабильному ID;
   потерянный ответ проверяется повторным чтением журнала.
9. Конкурирующие append и удаления сериализуются по таблице.
10. Исправление операции больше не удаляет корректную замену при потерянном
    ответе на удаление исходной записи.
11. Добавлены backup/recovery `state.json`, `/status`, Docker healthcheck и
    debounce частых синхронизаций `Money`.
12. После выбора базовой валюты бот предлагает создать первый счёт, а превышение
    дневного бюджета показывается явно.

## Выполненные проверки

- `npm run check`, `npm test`, `npm run build` — успешно;
- 10 тестовых файлов, 56 тестов — успешно;
- production `npm audit` — 0 известных уязвимостей;
- read-only smoke текущего подключения — Google Sheets отвечает, очередь
  `pending/failed` пуста, секреты и spreadsheet ID не выводились;
- Dockerfile дополнен healthcheck, но локальный Docker CLI в среде аудита не
  установлен, поэтому полноценная сборка образа здесь не запускалась.

## Сильные стороны, которые важно сохранить

- `Overview` не используется как финансовая база и не изменяется ботом.
- Расход, пополнение и перевод имеют разные финансовые смыслы.
- `Money` не загрязняется пополнениями и техническими полями.
- Отмена расхода связывает журнал и видимую строку через `tx_id`.
- Курсы операции сохраняются исторически.
- Ошибка зеркалирования в `Money` не теряет расход: есть статус и фоновый retry.
- Незавершённый Telegram-сценарий переживает перезапуск.

## Приоритетный план

### P0 — целостность денег

1. **Выполнено:** стабильные ID сохраняются в draft, append проверяет журнал до
   записи и после неопределённого ответа Google API.
2. **Выполнено:** финансовые append/delete защищены специализированными locks;
   перевод записывается и отменяется парой.
3. **Выполнено:** `state.json.bak`, структурная проверка, quarantine повреждённого
   JSON и восстановление состояния.
4. **Осталось проверить на копии реальной книги:** потерянный ответ API,
   двойное нажатие, restart между append журнала и записью `Money`, одновременный
   ручной ввод в `Money`.

Критерий готовности: любой повтор update или сетевого запроса создаёт не более
одной финансовой операции, а отмена никогда не оставляет активной половину
перевода.

### P1 — тесты и эксплуатация

1. Покрыть Telegram FSM интеграционными тестами: расход, пополнение, назад,
   отмена, stale callback, повторное нажатие, восстановление draft.
2. Вынести работу с Google API за тестируемый интерфейс и добавить контрактные
   тесты batch requests, диапазонов, notes и изменения Google Table.
3. **Выполнено:** admin-команда `/status`: активная поездка, время последней успешной
   синхронизации, число pending/failed, доступность Sheets API — без вывода
   секретов.
4. Перейти на структурированные логи с event name, spreadsheet hash, duration и
   error class. Не писать токены, полный spreadsheet ID и содержимое операций.
5. **Выполнено:** Docker healthcheck и документированный backup/restore для volume с
   `state.json`.
6. **Выполнено:** ручная синхронизация `Money` дебаунсится; открытие нескольких экранов
   подряд создаёт повторные чтения Google API и задерживает интерфейс.

Критерий готовности: критические сценарии воспроизводятся без Telegram и живой
таблицы, а владелец за одну минуту понимает, работает ли бот и есть ли очередь
ошибок.

### P2 — продуктовая полировка

1. Добавить пагинацию счетов, категорий и поездок, чтобы не упираться в лимиты
   inline-клавиатуры Telegram.
2. **Частично выполнено:** после валюты предлагается первый счёт; остаются шаг
   часового пояса и отменяемый тестовый расход.
3. **Выполнено:** превышенный дневной бюджет показывается отдельным
   предупреждением и фактической суммой превышения.
4. Добавить предупреждение о низком остатке и возможность временно отключить
   счёт без удаления истории.
5. Для голосового и естественного ввода показывать подтверждение перед записью,
   если модель не уверена в сумме, валюте, счёте или типе операции.
6. Добавить экспорт диагностического отчёта без персональных данных.

## Решение по дизайну

Не нужен отдельный Web UI: он продублирует Google Sheets и усложнит поддержку.
Лучшее направление — сохранить Telegram как «быстрый пульт», Google Sheets как
подробный архив и инвестировать в целостность операций, скорость панели и
диагностику. Визуальную часть следует улучшать внутри текущей системы: короткие
тексты, стабильная иерархия кнопок, прогресс, понятные подтверждения и
предсказуемый возврат назад.
